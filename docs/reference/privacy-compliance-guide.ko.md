# 개인정보 컴플라이언스 가이드

## 목적
- 빙고 서비스의 개인정보 처리 안내 및 개인정보처리방침에 대한 구현·운영 기준을 제공한다.
- 다음을 구분한다.
- 서비스 제공 / 계약 이행 근거로 처리 가능한 항목
- 별도 동의가 필요한 항목
- 제3자 제공에 가까운지, 처리위탁에 가까운지 판단이 필요한 항목

## 제품 기본 원칙
- 빙고 로그인 화면에서 서비스 필수 데이터를 전부 묶은 일괄 `필수 동의` 체크박스를 강제하지 않는다.
- 다음 두 가지를 분리해 제공해야 한다.
- DevFactory 플랫폼 개인정보처리방침 공개 페이지 (`/privacy` 같은 안정적인 경로)
- 로그인 전 행사 참가자 개인정보 처리 안내와 행사별 공개 페이지 (`/event/:slug/privacy` 같은 행사 경로)
- 플랫폼 공개 정책 페이지는 Google OAuth 동의 화면의 privacy policy URL로도 사용할 수 있을 만큼 안정적인 경로여야 한다.

## 개인정보처리방침 최소 구조
- 개인정보처리자 또는 운영 주체와 적용 범위를 식별한다.
- 개인정보 항목과 법적 근거를 설명할 때, 동의 없이 처리하는 항목과 동의를 받아 처리하는 항목을 구분한다.
- 다음 항목을 포함한다.
- 처리 목적
- 개인정보 항목 범주
- 보유 기간 또는 보유 기준
- 해당 시 제3자 제공 내용
- 해당 시 처리위탁 내용
- 해당 시 국외 이전 내용
- 정보주체 권리와 연락 방법
- 파기 절차
- 안전성 확보조치 개요
- 방침 변경 고지 방법

## 별도 동의 판단표
| 상황 | 별도 동의 | 메모 |
|---|---|---|
| 로그인, 참가자 식별, 보드 생성, 키워드 교환, 진행률 계산 | 보통 불필요 | 서비스 제공에 필요한 최소 범위라면 명확한 고지를 전제로 처리 가능 |
| 사용자가 직접 남긴 평점/후기를 행사 결과 흐름 안에서 사용하는 경우 | 보통 불필요 | 목적, 보유기간, 노출 범위를 명확히 고지해야 함 |
| 동일 행사 또는 동일 서비스 운영을 위한 내부 집계 통계 | 보통 불필요 | 고지된 목적 범위 안에서만 사용 |
| 행사 간 프로파일링, 추천, 마케팅 대상 구축 | 필요 또는 강력 권장 | 최소 서비스 운영 범위를 넘어감 |
| 스폰서 공유, 파트너 전달, 주최측의 독자적 활용 | 보통 필요 | 제3자 제공에 해당할 가능성이 큼 |
| 민감정보 수집 | 필요 | 별도 분리하여 보수적으로 처리 |
| 법정 연령 기준 검토가 필요한 아동 정보 처리 | 필요 | 법정대리인 동의 이슈 별도 검토 |

## 제3자 제공 vs 처리위탁 판단표
| 상황 | 기본 분류 | 이유 |
|---|---|---|
| 동일 개인정보처리자 조직 내부에서 권한을 가진 Admin/Event Manager가 처리 | 내부 처리 | 개인정보 지배·관리권이 외부로 이전되지 않음 |
| 별도 행사 주최측이 참가자 데이터를 자신의 독자적 운영 또는 마케팅에 사용 | 제3자 제공 가능성 큼 | 다른 법인·단체로 통제와 이익이 이동함 |
| 클라우드 DB, 인증, 이메일, 인프라 업체가 우리 지시에 따라 처리 | 보통 처리위탁 | 개인정보처리자를 대신해 처리 |
| 한국 외 지역에서 서비스 데이터가 저장·처리되는 외부 업체 사용 | 처리위탁 + 국외 이전 검토 | 추가 고지 또는 이전 근거 검토가 필요할 수 있음 |
| Google Sign-In으로 프로필 정보가 앱으로 전달되는 경우 | 외부 인증 연동 | Google 동의는 Google 계정 접근 범위에 대한 것이며, 서비스 내부 처리 전부를 대체하지 않음 |

## 이벤트 빙고 전용 점검 체크리스트
- 행사별 실제 개인정보처리자가 누구인지 확인한다.
- 플랫폼 운영자 단독인지
- 행사 주최측 단독인지
- 공동 또는 분리 역할인지
- 플랫폼과 행사 주최측의 역할이 다르면 한 문안에 섞지 말고 별도 안내로 분리한다.
- 관리자 콘솔의 Event Manager가 내부 인력인지 외부 주최측인지 확인한다.
- 참가자 이름, 이메일, 키워드, 진행률 정보가 별도 주최측에 제공되는지 확인한다.
- Supabase 리전, 이메일 발송, 분석, 스토리지 등으로 국외 이전 검토가 필요한지 확인한다.
- 핵심 행사 운영 외의 선택 목적이 존재하는지 확인한다.
- 선택 목적이 존재하면 다음을 추가한다.
- 별도 체크박스 문구
- 동의 로그
- 버전 관리되는 동의 문안
- 필수 서비스 데이터와 분리된 거부 경로

## 행사 아카이브 보관 패턴
- 행사 메타데이터와 익명·집계 통계를 직접 식별정보와 분리해서 관리한다.
- 일회성 행사이고 장기 사후관리가 없다면 다음 기준을 우선 검토한다.
- 이름, 이메일, 로그인 식별자, 행사별 표시 이름, 후기 원문 같은 직접 식별정보는 행사 종료 후 1년 이내 삭제 또는 익명화
- 참가자 수, 빙고 완성 수, 키워드 교환 건수, 평균 평점, 행사 소개 정보 등 익명·집계 정보만 장기 보관
- 내부 `user_id`처럼 서비스가 다시 사람과 연결할 수 있는 값은 그대로 남겨두면 완전한 익명정보로 보기 어렵다.

## Google OAuth 운영
- Google Cloud OAuth consent screen에 프로덕션 홈페이지 URL과 프로덕션 `/privacy` URL을 유지한다.
- DevFactory 플랫폼 방침과 행사 참가자 안내가 겹치는 범위는 서로 모순되지 않게 유지한다.
- 요청 scope 또는 Google 데이터 사용 방식이 바뀌면 공개 정책 페이지와 인앱 안내를 함께 갱신한다.

## 구현 메모
- 코드의 기본 방침 문안은 보수적으로 작성하고, 운영에서 입증할 수 없는 제3자 제공 또는 무제공 사실을 단정하지 않는다.
- 배포 환경마다 달라지는 사실은 게시 전에 Admin이 확인해야 한다.
- 과거의 `동의 완료` 플래그만으로는 별도 선택 동의 증거로 사용하지 않는다. 별도 동의로 쓰려면 문안, 버전, 시각, 사용자의 동작 경로가 함께 기록되어야 한다.

## 현재 저장소 기준 확인 사항 (2026-04-15)
- 운영주체:
- 저장소에서 보이는 서비스 표기는 `PseudoLab` / `DevFactory` 기준이다 (`frontend/src/config/eventProfiles.ts`, `frontend/src/modules/Bingo/BingoGame.tsx`, `frontend/LICENSE`).
- 다만 앱 설정 어딘가에 단일한 법인명 또는 최종 개인정보처리자 명칭이 중앙 관리되고 있지는 않다.
- 처리위탁:
- 현재 프로덕션용 초안은 `Supabase`를 인증·DB 처리 수탁자로, SMTP 이메일 발송 사업자를 관리자 초대/알림 메일 수탁자로 전제한다.
- 다만 저장소에는 정확한 SMTP 사업자명이 없으므로, 공개 처리방침에는 실제 계약한 사업자명으로 대괄호 placeholder를 교체해야 한다.
- 국외이전:
- 환경 예시에 `aws-1-ap-southeast-2.pooler.supabase.com` 호스트가 포함되어 있어, 프로덕션도 같은 구조라면 Supabase 실제 리전 국가/도시를 기재한 국외 이전 고지가 필요하다.
- SMTP 사업자가 국외 사업자이면 해당 국가/도시와 이전 항목·목적도 함께 기재해야 한다.
- 보유기간:
- 코드베이스에는 행사 초기화/삭제 같은 수동 삭제 흐름은 있지만, 자동 삭제 스케줄러나 보유기간 집행 배치는 없다.
- 다만 만료된 행사 개인정보를 익명화하는 startup 옵션 기반 헬퍼는 추가되어 있으므로, 운영에서 이 옵션을 실제로 켜고 결과를 검증해야 한다.
- 따라서 공개 처리방침의 보유기간은 코드 기본값이 아니라 실제 운영 절차와 일치해야 한다.
