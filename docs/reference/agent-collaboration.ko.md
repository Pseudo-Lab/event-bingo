## 핵심 규칙
- 동일한 기준 문서에서 동일한 결과를 내야 한다.
- 모델별 숨은 메모리에 의존하지 않는다.
- 가정 사항은 핸드오프 노트에 반드시 명시한다.
- 구현은 재현 가능하고 테스트 가능해야 한다.

## 배포 레포지토리 경계
- 이 저장소는 애플리케이션 저장소다.
- k3s/ArgoCD GitOps 매니페스트는 `https://github.com/Pseudo-Lab/DevFactory-Ops`에서 관리한다.
- 배포/런타임 토폴로지 변경이 필요하면 GitOps 저장소 연계 핸드오프를 함께 남긴다.

## 도메인 소유 및 머지 정책
- BE 도메인 소유 범위: `backend/**`
- FE 도메인 소유 범위: `frontend/**`
- Infra(lead) 도메인 소유 범위: 이 저장소의 `.github/workflows/**`, `docker-compose.yaml`, `k8s/**`와 `Pseudo-Lab/DevFactory-Ops`의 배포/운영 매니페스트
- 도메인 소유자는 단일 도메인, 비영향 변경에 한해 PR 없이 `main` 직접 푸시할 수 있다.

## 영향 트리거 (PR 필수)
- 여러 도메인 파일을 동시에 변경한다.
- API 계약, 인증 동작, DB 스키마, 마이그레이션이 다른 도메인에 영향을 준다.
- `docs/*.md` 기준 문서를 변경한다.
- CI/CD, 배포, 보안, ingress, secrets, 런타임 토폴로지 변경이 포함된다.
- `Pseudo-Lab/DevFactory-Ops` 연계 변경이 필요한 경우가 포함된다.
- Product Owner 또는 해당 도메인 소유자가 PR 리뷰를 요청한다.

## 핸드오프 필수 정책
핸드오프 노트는 영향 트리거 작업에서 필수이며, 중간 이상 규모 작업에서는 강하게 권장한다.
핸드오프 노트는 저위험, 소규모, 단일 도메인 변경에서 커밋 또는 PR 설명만으로 맥락이 충분하면 생략할 수 있다.

## 팀 리드 모드
팀 리드 모드는 선택 사항이며, 사용자가 팀 방식 또는 멀티 에이전트 작업을 명시적으로 요청한 경우에만 사용한다.
소규모 단일 도메인 작업에는 팀 리드 모드를 자동으로 켜지 않는다.

팀 리드 에이전트는 모든 구현 세부사항을 직접 소유하는 것이 아니라 조율을 책임진다.
팀 리드 책임:
- 작업 목표, 포함 범위, 제외 범위, 완료 기준을 다시 명확히 한다.
- 작업에 필요한 최소 기준 문서만 읽는다.
- 작업을 배정하기 전에 현재 워크트리 상태를 확인한다.
- 역할과 파일 소유권 기준으로 작업을 나눈다.
- 병렬로 안전하게 진행할 수 있는 작업과 순차로 진행해야 하는 작업을 구분한다.
- 하위 에이전트에게 명확한 소유 범위가 있는 제한된 작업을 배정한다.
- 통합 전에 하위 에이전트 결과를 검토한다.
- 충돌은 기준 문서 우선순위, Product Owner 의도, 도메인 소유자의 기술 판단 순서로 해결한다.
- 최종 통합 요약, 검증 근거, 후속 작업 목록을 작성한다.

## 팀 역할 및 소유권
사용 가능한 팀 역할은 매 작업마다 반드시 투입해야 하는 참여자가 아니다.
팀 리드는 작업에 필요한 최소 역할만 활성화해야 한다.

핵심 역할:
- `product-owner`: 범위, 인수 기준, 순서, 릴리즈 리스크, 트레이드오프 결정
- `backend-api`: `backend/**`, API 계약, 스키마, 서비스, 모델, 마이그레이션, 백엔드 테스트
- `frontend`: `frontend/**`, UI 흐름, 프론트 API 클라이언트, 상태 처리, 반응형 동작, 프론트 테스트
- `qa`: 테스트 계획, 읽기 전용 감사, 결함 재현, 타깃 회귀 검증, 릴리즈 품질 요약

조건부 역할:
- `design`: 디자인 가이드 준수, UX 검토, 시각적 일관성, 접근성 및 반응형 레이아웃 검토. UI/UX 품질, 시각 디자인, 반응형 동작이 주요 관심사일 때 사용한다.
- `security`: 인증/인가 검토, 개인정보성 데이터 흐름, secrets/config 노출, 공개/비공개 API 경계, export, 보관 정책, 배포 보안 리스크. 인증, 인가, 개인정보성 데이터, export, secrets, 런타임 설정, API 접근 경계, 배포 보안에 영향을 주는 작업에서만 사용한다.
- `infra`: `.github/workflows/**`, `docker-compose*.yaml`, `k8s/**`, 배포/런타임 토폴로지 노트

소유권 규칙:
- 하나의 파일은 한 번에 하나의 활성 편집 소유자만 가져야 한다.
- `frontend/src/modules/Admin/AdminPortal.tsx`, `frontend/src/modules/Bingo/BingoGame.tsx`처럼 큰 공유 파일은 여러 역할이 동시에 편집하지 않는다.
- API 계약 변경은 `backend-api`, `frontend`, `qa` 사이에 조율해야 한다.
- 기준 가이드 문서 변경은 같은 변경 세트에서 한국어 미러를 함께 업데이트해야 한다.

## 병렬 작업 규칙
쓰기 범위가 분리되어 있고 계약이 명확한 경우 병렬 작업을 허용한다.
좋은 병렬 후보:
- API 계약이 이미 안정된 상태에서 백엔드 라우트/서비스 변경과 프론트 디자인 탐색을 병렬 진행
- 구현 에이전트가 코드를 수정하는 동안 QA가 읽기 전용 테스트 감사를 진행
- 구현 에이전트가 인증, 개인정보, API, export, 설정, 배포 보안 표면을 수정하는 동안 security가 읽기 전용 리뷰를 병렬 진행
- 백엔드/프론트 구현이 진행되는 동안 Product Owner가 인수 기준을 정리
- 기준 요구사항을 바꾸지 않는 문서 또는 핸드오프 초안 작성

순차 작업이 필요한 경우:
- 한 작업이 아직 확정되지 않은 API 계약에 의존한다.
- 여러 역할이 같은 파일을 편집해야 한다.
- 마이그레이션 또는 스키마 변경이 프론트 요청/응답 처리에 영향을 준다.
- QA가 구현 안정화 후 최종 동작을 검증해야 한다.
- 개인정보, 보안, 배포 토폴로지 등 영향 트리거 표면을 건드린다.

## 하위 에이전트 보고 계약
각 하위 에이전트는 아래 내용을 보고해야 한다.
- 역할과 배정 범위
- 읽은 파일
- 변경한 파일
- 동작 변경
- 실행한 검증과 결과
- 알려진 리스크 또는 블로커
- 필요한 후속 작업

하위 에이전트 보고는 그 자체로 최종 결과가 아니다.
팀 리드는 이를 검토하고 하나의 최종 답변 또는 핸드오프 노트로 통합해야 한다.

## QA/테스트 투입 시점
QA 에이전트는 팀 리드가 좁은 테스트 수정 범위를 배정하지 않는 한 읽기 전용으로 시작한다.
다른 에이전트가 소스 코드를 활발히 수정 중일 때 QA는 아래 작업을 우선한다.
- 테스트 목록화
- 커버리지 및 리스크 감사
- 재현 가능한 시나리오 설계
- 알려진 실패 문서화

QA가 테스트를 수정할 수 있는 조건:
- 해당 동작의 구현 작업이 안정화되어 있다.
- 파일 소유권이 배정되어 있다.
- 기대 동작이 명확하다.
- 변경 범위가 병합 충돌을 피할 수 있을 만큼 좁다.

## 보안 리뷰 투입 시점
Security 에이전트는 팀 리드가 좁은 보안 수정 범위를 배정하지 않는 한 읽기 전용으로 시작한다.
인증, 인가, 개인정보성 데이터, export, secrets, 런타임 설정, 공개/비공개 API 경계, 배포 보안에 영향을 주는 보안 수정은 영향받는 역할과 조율해야 한다.
security와 infra 관심사가 겹칠 때는 아래처럼 소유권을 나눈다.
- `infra`: 런타임 연결, 배포 토폴로지, CI/CD, 컨테이너, 환경변수 주입, GitOps 조율을 소유한다.
- `security`: 노출, 접근 제어, 데이터 보호, secret 처리, 악용 또는 유출 경로에 대한 리스크 검토를 소유한다.

## 경량 완료 보고 규칙
핸드오프 노트를 생략하는 경우에는 커밋 또는 PR 설명에 아래 내용을 짧게 남긴다.
- 무엇을 변경했는지
- 어떻게 검증했는지
- 다음 영향 또는 후속 작업이 있는지

## 필수 핸드오프 작업 계약 항목
필수 핸드오프 노트에는 아래 항목이 있어야 한다.
- Task ID
- Scope
- Inputs used
- Outputs created or changed
- Risks
- Next owner

템플릿: `docs/templates/agent-handoff.md`

## 브랜치/커밋 규칙
- direct push 경로: 단일 도메인, 비영향 작업은 `main`에 직접 커밋한다.
- PR 경로 브랜치: `feature/<task-id>-<short-slug>` 또는 `fix/<task-id>-<short-slug>` 또는 `codex/<task-id>-<short-slug>`
- 커밋 제목: `<type>(<scope>): <summary>`
- 커밋 본문 메타데이터는 선택 사항이며, 커밋 메시지는 변경 의도를 중심으로 간결하게 유지한다.

## PR 협업 체크리스트
- 기준 문서 참조 여부
- 영향 트리거 체크 및 사유 명시 여부
- 영어 가이드 변경 시 한국어 미러 동기화 여부
- 테스트 근거 첨부 여부
- 핸드오프 템플릿 첨부 여부

## 충돌 해결
- 에이전트별 결과가 다르면 문서 우선순위 기준으로 비교한다.
- 제품 의도 충돌은 Product Owner가 결정한다.
- 기술적 충돌은 해당 도메인 소유자가 결정한다.

## 언어 및 동기화 규칙
- 구현 기준은 영어 가이드 문서다.
- 한국어 미러는 같은 변경에서 반드시 동기화한다.
- 본 문서 동기화 페어:
- `docs/reference/agent-collaboration.md` <-> `docs/reference/agent-collaboration.ko.md`
