# 서비스 유저 플로우

## 확정된 흐름
1. 사용자가 로그인한다.
2. 사용자가 관심 키워드를 선택한다.
3. 오프라인에서 다른 참가자를 만나, 상대 입력창에 자신의 코드를 입력한다.
4. 보낸 사람의 키워드가 받는 사람의 빙고판에 반영된다.
5. 개인전/팀전 진행 상태가 실시간으로 갱신된다.

## 핵심 규칙
- 빙고판 셀은 행사별 첫 로그인 시 사용자 단위로 랜덤 생성한다.
- 최초 생성된 빙고판 배치는 이후 고정한다.
- 빙고판 키워드 배정은 최초 생성 이후 고정한다.
- 교환 동작은 개인전과 팀전을 모두 지원해야 한다.
- 행사는 기본적으로 설정된 시작 시간 전 참가자 입장을 차단한다.
- 이벤트 관리자는 리허설 또는 사전 입장을 위해 시작 전 참가 제한을 해제할 수 있으며, 해제 시 참가자는 행사 시작 전에도 입장하고 빙고판을 만들 수 있다.

## 서비스 화면 구성
1. 로그인 화면
- 구성된 로그인 방식의 진입 CTA
- 로그인 전 행사 참가자 개인정보 처리 안내 링크
- DevFactory 플랫폼 개인정보처리방침 페이지 접근

2. 키워드 선택 화면
- 지정 개수만큼 키워드 선택
- 시작 버튼으로 게임 화면 진입

3. 게임 화면
- 프로필 영역 및 키워드 전송 입력
- 개인 완료율 및 카운터
- 빙고판 시각화 및 완성 라인 표시
- 상호작용 리스트 2개:
- 내가 준 사람 목록
- 나에게 보낸 사람 목록

4. 팀전 패널
- 팀 대 팀 상태 카드
- 목표 진행 바 및 남은 줄 수
- 시간 종료/목표 달성 종료 상태

## 데이터 및 이벤트 노트
- 교환 이벤트는 sender, receiver, event_id, timestamp 추적 가능해야 한다.
- 보드 반영과 교환 히스토리의 불일치가 발생하지 않도록 보장해야 한다.

## 개발 테스트 모드
- `?testMode=1`은 개발 및 private preview 확인을 위한 프론트엔드 전용 도구다.
- 테스트 모드는 행사 시간 잠금을 우회하고 빙고판 프리뷰 컨트롤을 노출할 수 있다.
- 운영 빌드는 명시적인 비운영 환경 플래그가 허용하지 않는 한 공개 URL의 테스트 모드 활성화를 무시해야 한다.
- 후속 권장 작업: 공개 query 활성화는 `VITE_ALLOW_PUBLIC_TEST_MODE=true` 같은 환경변수 뒤에 두고, 운영 기본값은 false로 유지하며, 백엔드 관리자 테스트 API는 계속 관리자 인증 뒤에 둔다.

## 언어 및 동기화 규칙
- 구현 기준은 영어 문서 `docs/reference/service-user-flow.md`를 우선 사용한다.
- 영어 문서 수정 시 본 한국어 문서도 반드시 함께 갱신한다.
